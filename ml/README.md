# AuxLess ML Pipeline

## Overview

The ML pipeline generates real-time group music recommendations for multi-user party sessions. It takes preprocessed playlist data from the batch pipeline, builds song embeddings, and runs a hybrid recommendation engine combining Content-Based Filtering (CBF), Collaborative Filtering (CF), and a GRU sequential model. Recommendations adapt in real time based on user feedback from the streaming pipeline.

Fully deployed on GCP with Cloud Run, BigQuery, Firestore, and Pub/Sub.

## Architecture

```
Batch Pipeline Completes
    |
    v
SESSION_READY_TOPIC (Pub/Sub) --> Cloud Run /ml endpoint
    |
    v
ml_trigger.py: _run_ml_session(session_id)
    |
    +-- Phase 1: Preprocess mitigated CSV, generate 386-dim embeddings
    +-- Phase 2: Load song catalog (81K songs), GRU model, session users
    +-- Phase 3: Generate first 30 recommendations (CBF + CF + GRU hybrid)
    +-- Phase 4: Background watcher regenerates 30 songs every 10 played
```

Data flows from GCS (mitigated CSV) through preprocessing and embedding generation into BigQuery (song catalog), then the recommendation engine reads from BigQuery and Firestore to produce ranked songs written back to Firestore for the frontend.

## Data Loading

The ML pipeline picks up where the batch pipeline leaves off. When the batch pipeline finishes, it publishes a session-ready event to Pub/Sub, which triggers the ML pipeline automatically.

Input: `gs://{bucket}/sessions/{session_id}/bias_mitigation/{session_id}_mitigated.csv`

This CSV has already gone through validation, bias detection, and bias mitigation in the batch pipeline.

Preprocessing (`ml/preprocessing/preprocess_songs.py`) deduplicates by video_id, fills nulls, cleans text (lowercase, strip special characters), coerces numeric fields, computes `duration_sec` and `popularity_score` (weighted combination of view, like, and comment counts via MinMaxScaler).

Output: `gs://{bucket}/ml_output/{session_id}_preprocessed.csv`

## Feature Engineering

Each song is represented as a 386-dimensional embedding vector.

The text component (384 dims) is generated by encoding `"title: X artist: Y genre: Z album: W"` through `sentence-transformers/all-MiniLM-L6-v2`. This captures semantic similarity between songs based on their metadata.

The numeric component (2 dims) consists of `duration_sec` and `popularity_score`, both MinMaxScaled to [0, 1].

These are concatenated to form the final 386-dim embedding. New songs from each session are uploaded to BigQuery, skipping duplicates. The current catalog has 81,449 songs.

Code: `ml/feature_engineering/generate_song_embeddings.py`, `ml/feature_engineering/bigquery_loader.py`

## Model Architecture

SessionGRU (`ml/gru/gru_model.py`) is a GRU-based sequential model that predicts the next song embedding from a sequence of played songs.

```
Input (batch, seq_len, 387)    // 386 embedding + 1 liked_flag
  -> Input Projection (387 -> 256, LayerNorm, ReLU, Dropout)
  -> GRU (2 layers, hidden=256, dropout=0.3)
  -> Last Hidden State (256)
  -> Output Projection (256 -> 386)
  -> L2 Normalize
  -> Predicted Embedding (386)
```

Total parameters: 1,054,338. At inference, the predicted embedding is compared against all catalog songs via cosine similarity to rank candidates.

## Training

Since AuxLess is a new system without historical data, we generate genre-aware synthetic sessions for training (`ml/gru/synthetic_data.py`). 5,000 sessions are created from the BigQuery catalog where each session has a preferred genre. 60% of songs match the preferred genre (liked at 85% probability) and 40% are from other genres (liked at 20% probability). Session length ranges from 5 to 15 songs.

For each session `[s1, s2, s3, ..., sN]`, multiple subsequence samples are created: `[s1]->s2`, `[s1,s2]->s3`, etc. Sequences are left-padded to length 15.

Training uses CosineEmbeddingLoss with AdamW optimizer (lr=1e-3, weight_decay=1e-4), ReduceLROnPlateau scheduler (patience=3), gradient clipping at 1.0, and early stopping (patience=7). Train/val split is 90/10.

Current best model: 21 epochs, validation loss = 0.3226. Stored at `gs://youtube-pipeline-staging-bucket/models/gru_model.pt`.

## Hyperparameter Tuning

Optuna searches over hidden_dim (128/256/512), num_layers (1-3), dropout (0.1-0.5), learning_rate (1e-4 to 1e-2, log scale), and batch_size (32/64/128). A MedianPruner stops underperforming trials early.

Data is loaded once and shared across all trials. After the search completes, the final model is retrained with the best hyperparameters and saved.

```bash
python -m ml.gru.train_gru --mode tune --sessions 5000 --trials 20
python -m ml.gru.train_gru --mode standalone --sessions 5000
```

## Experiment Tracking

All training runs are tracked with MLflow under the experiment `auxless-gru-training`. Each run logs hyperparameters, per-epoch train/val loss, learning rate, final metrics (best val_loss, total epochs, early stopping status), and the model artifact via `mlflow.pytorch.log_model()`.

Optuna trials are logged as nested runs under a parent study run, making it easy to compare all trials in the MLflow UI.

```bash
mlflow ui  # view at http://localhost:5000
```

## Model Validation

Validation happens at two stages. During training, a 10% hold-out validation set is used to track CosineEmbeddingLoss per epoch. Early stopping and learning rate scheduling both monitor validation loss. The best model is selected based on the lowest val_loss across all epochs.

During inference, GRU predictions are L2-normalized (valid cosine similarity range [-1, 1]), already-played songs are excluded, and recommendations are validated before writing to Firestore.

## Bias Detection

Bias is evaluated at two levels.

**Data level** (batch pipeline): `bias_analyser.py` computes genre and country distributions, Shannon entropy, dominant label share, and flags slices exceeding 60%. `bias_mitigation.py` upsamples slices below 5% and downsamples slices above 60%, with a report documenting before/after proportions and trade-offs.

**Model level** (ML pipeline): `ml/evaluation/model_bias.py` checks three dimensions on the recommendation output:

1. Genre representation: compares genre distribution in recommendations vs the catalog. Flags over-represented genres (>40%) and under-represented genres (<2%).
2. Popularity bias: compares average popularity of recommended songs vs catalog average. Flags when recommendations are >1.5x more popular.
3. Score disparity: checks if CBF/GRU scores are consistent across genres. Flags when max score difference between genres exceeds 0.15.

Each check produces mitigation suggestions (genre diversity constraints, popularity penalties, per-genre score normalization).

## Sensitivity Analysis

Optuna trial results show that learning rate is the most sensitive parameter, with values in 1e-3 to 5e-3 producing the lowest validation loss. Hidden dimension of 256 performs best (128 underfits, 512 overfits on synthetic data). Two GRU layers are optimal. Dropout of 0.3 balances regularization and capacity.

The 386-dim embedding captures feature importance implicitly: text embeddings (384 dims) dominate the representation, while numeric features (2 dims) provide supplementary duration and popularity signals. The liked_flag (1 dim, GRU input only) lets the model weight liked songs more heavily in sequential predictions.

The hybrid weights (CBF 0.4, CF 0.3, GRU 0.3) control the balance between group preferences, population co-occurrence patterns, and sequential patterns. The additive session score (0.3) ensures real-time feedback always has influence.

## Recommendation Engine

The engine combines three signals in a hybrid architecture.

**CBF** (`ml/recommendation/CBF.py`) builds a global group preference vector by averaging per-user vectors (liked embeddings minus 0.5 times disliked embeddings, L2 normalized, weighted by liked count). Cosine similarity against all catalog songs produces the base ranking, nudged by real-time session feedback scores.

**CF** (`ml/recommendation/cf/CF.py`) uses item-item co-occurrence across the full user population. For each candidate song, it finds the strongest connection between that song and any of the group's liked songs based on how many users in the database liked both. Scores are normalized by the candidate's total popularity to avoid surfacing songs that score high simply because everyone has them. When population data is unavailable (cold deployment), CF falls back to embedding-based similarity between candidates and the group's liked songs, producing scores in the same [0, 1] range so the aggregation requires no special handling.

**GRU** (`ml/gru/gru_inference.py`) activates after 3 songs have been played. It takes the play sequence (embedding + liked flag per song), runs a forward pass to predict the next song embedding, and ranks candidates by cosine similarity.

Weights ramp gradually:

| Songs Played | CBF | CF | GRU | Strategy |
|---|---|---|---|---|
| 1-3 | 0.6 | 0.4 | 0.0 | Cold start |
| 4 | 0.4 | 0.3 | 0.2 | GRU soft entry |
| 5 | 0.4 | 0.3 | 0.25 | GRU growing |
| 6+ | 0.4 | 0.3 | 0.3 | Full hybrid |

Feedback scoring: like (+2.0), replay (+1.5), skip (-0.5), dislike (-2.0). Scores are normalized to [-1, +1] before application.

## Continuous Training & Monitoring (CT/CM)

AuxLess implements a full CT/CM pipeline to detect data drift and automatically retrain the GRU model as real session data accumulates.

### Data Drift Detection (`ml/ct_cm/drift_detector.py`)

Drift is detected by comparing the genre distribution of real production sessions against the training genre distribution using KL divergence.

**Training distribution** reflects the genre mix in the BigQuery catalog used during synthetic GRU training (chill: 15%, energize: 12%, commute: 11%, ...).

**Production distribution** is computed from Firestore `song_events` across all completed sessions in the last 7 days.

**KL divergence** measures how much the production distribution has shifted:
- KL = 0.0 → identical distributions, no drift
- KL > 0.3 → significant drift, retraining recommended

Current production KL = 1.2844 (drift detected — users are listening to more energize and commute songs than the training distribution expected).

Results are written to Firestore `drift_events` and published to Cloud Monitoring as `custom.googleapis.com/auxless/genre_drift_kl`.

### Continuous Training (`ml/ct_cm/retrain.py`)

When triggered, the CT pipeline:

1. Fetches real play sequences from Firestore `song_events` (completed sessions, 5+ songs)
2. Builds training samples using the same subsequence format as synthetic training
3. Splits 90/10 train/val and retrains GRU using `train()` from `train_gru.py`
4. Compares new val_loss against production model val_loss (from GCS `retrain_history.json`)
5. Deploys new model to GCS only if improvement > 0.01 (1%) — prevents regression
6. Backs up current model before deploying
7. Publishes val_loss to Cloud Monitoring as `custom.googleapis.com/auxless/gru_val_loss`
8. Writes result to Firestore `retrain_events`

Minimum 50 real sessions required before retraining is attempted. Currently skipping — 9 sessions available.

## Model Versioning and Rollback

The trained model is stored at `gs://youtube-pipeline-staging-bucket/models/gru_model.pt`. MLflow logs every training run with the full model artifact, hyperparameters, and validation metrics. GCS object versioning supports rollback to previous model versions.

If the model fails to load at session start, the system gracefully falls back to CBF + CF only recommendations. This ensures the service never goes down due to a bad model.
